# Liquibase Guide

## The Golden Rule

**Never change a changelog file after it has been committed to git.**

Liquibase fingerprints (checksums) every changeset the first time it runs. If the file changes, the checksum changes, and Liquibase refuses to start:

```
Validation Failed: 1 change sets check sum was changed
```

This means: no editing old changelogs, no deleting them, no reordering `master.xml`. Think of changelogs like git commits — once pushed, they're immutable.

This includes `00000000000001_custom_schema.xml` — it is **not** a "sequences file" you can keep adding to. It is a frozen changelog like any other.

## How to Make Database Changes

### Preferred: JDL workflow

JHipster generates incremental Liquibase changelogs automatically. It will NOT touch existing changelog files.

**Step 1:** Edit `jdl.jdl` — add or change fields:

```jdl
entity Product {
  // ... existing fields ...
  newField String    // <-- add your field here
}
```

**Step 2:** Run the generator:

```bash
npx jhipster jdl jdl.jdl --force --skip-install
```

**Step 3:** Review what was generated:

```bash
git diff --name-only
```

You should see:

- A **new** changelog file like `TIMESTAMP_added_entity_Product.xml` with `createTable` (new entity) or `TIMESTAMP_updated_entity_Product.xml` with `addColumn` (existing entity)
- `master.xml` with a new include line
- `.jhipster/Product.json` updated
- Java and TypeScript source files updated

You should NOT see changes to any existing changelog.

**Step 4: Add the sequence (new entities only).**

JHipster does not generate `createSequence` statements. Without one, Hibernate will fail at runtime when inserting rows. Add it manually as the **first changeset** in the generated entity changelog:

```xml
<!-- In TIMESTAMP_added_entity_YourEntity.xml, BEFORE the createTable changeset -->
<changeSet id="TIMESTAMP-seq" author="yourname">
    <createSequence sequenceName="sequence_generator_your_entity" startValue="1" incrementBy="1"/>
</changeSet>

<!-- This changeset is already generated by JHipster — leave it as-is -->
<changeSet id="TIMESTAMP-1" author="jhipster">
    <createTable tableName="your_entity">
        <!-- columns -->
    </createTable>
</changeSet>
```

Do **NOT** add the sequence to `00000000000001_custom_schema.xml`. That file is frozen.

**Step 5:** Review the generated changelog, then commit.

**Rules:**

- Always use `jhipster jdl`, not `jhipster entity` (the entity command doesn't handle incremental changelogs correctly)
- Don't edit `.jhipster/*.json` files before running the generator — JHipster compares old vs new to generate the diff
- Review generated changelogs before committing — if one looks wrong, delete it and write it manually

### Manual: Write XML yourself

For changes that JDL can't express (custom constraints, data migrations, stored procedures), write the changelog by hand.

**1.** Create a new file with a timestamp name:

```bash
touch src/main/resources/config/liquibase/changelog/20251230100000_add_status_to_product.xml
```

**2.** Write the changeset:

```xml
<?xml version="1.0" encoding="utf-8"?>
<databaseChangeLog xmlns="http://www.liquibase.org/xml/ns/dbchangelog"
                   xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
                   xsi:schemaLocation="http://www.liquibase.org/xml/ns/dbchangelog
                   http://www.liquibase.org/xml/ns/dbchangelog/dbchangelog-latest.xsd">

    <changeSet id="20251230100000-1" author="yourname">
        <addColumn tableName="product">
            <column name="status" type="varchar(50)"/>
        </addColumn>
    </changeSet>
</databaseChangeLog>
```

**3.** Add one line at the **end** of `master.xml`:

```xml
<include file="config/liquibase/changelog/20251230100000_add_status_to_product.xml" relativeToChangelogFile="false"/>
```

### Quick reference for manual changesets

```xml
<!-- Change a column type -->
<modifyDataType tableName="product" columnName="description" newDataType="varchar(1000)"/>

<!-- Drop a column -->
<dropColumn tableName="tariff" columnName="grid_operator_id"/>

<!-- Add a foreign key -->
<addForeignKeyConstraint baseTableName="product_offering_price" baseColumnNames="tariff_id"
    constraintName="fk_pop_tariff" referencedTableName="tariff" referencedColumnNames="id"/>

<!-- Drop a table (drop FKs first) -->
<dropForeignKeyConstraint baseTableName="other_table" constraintName="fk_other_to_mytable"/>
<dropTable tableName="my_table"/>
```

### Stored procedures and views

These are safe to update in place because `CREATE OR REPLACE` is idempotent. Add `runOnChange="true"`:

```xml
<changeSet author="developer" id="create-my-procedure" runOnChange="true">
    <createProcedure>
        CREATE OR REPLACE PROCEDURE my_procedure AS BEGIN ... END;
    </createProcedure>
</changeSet>
```

Do NOT use `runOnChange` for tables, columns, or foreign keys.

## Checklist Before You Commit

- [ ] No existing changelog files were modified (especially `00000000000001_custom_schema.xml`)
- [ ] New changelog has a unique timestamp-based ID
- [ ] New entity has a `createSequence` changeset in its own changelog (not in `custom_schema.xml`)
- [ ] New lines added at the END of `master.xml`
- [ ] Tested locally with `./mvnw spring-boot:run` (Liquibase runs on startup)

## Fixing Checksum Errors

### Development: Start fresh

Wipe your local database and let Liquibase rebuild from scratch:

```bash
docker compose -f src/main/docker/services.yml down -v
docker compose -f src/main/docker/services.yml up -d
```

### Production: Add the old checksum

Add a `validCheckSum` with the original fingerprint. Liquibase will accept both:

```xml
<changeSet id="20250313145035-1" author="developer">
    <validCheckSum>8:b4fd16a20425fe377b00d81df722d604</validCheckSum>
    <!-- rest of changeset -->
</changeSet>
```

Find the old checksum in the error message or query the database:

```sql
SELECT id, md5sum FROM databasechangelog WHERE id = 'your-changeset-id';
```

## Common Questions

### "I made a mistake in a changelog I just created"

| Situation                     | What to do                                       |
| ----------------------------- | ------------------------------------------------ |
| Not pushed to git yet         | You can still edit it                            |
| Pushed, but no one ran it yet | Talk to your team, you may be able to force-push |
| Someone already ran it        | Create a NEW changelog to fix the problem        |

### "JHipster overwrote my changelog files"

This shouldn't happen with the JDL workflow. If it does, someone used `jhipster entity` directly:

1. `git checkout -- src/main/resources/config/liquibase/changelog/` to restore
2. Use `jhipster jdl` instead

## Fake Data (Faker / Dev Seed Data)

Fake data changesets run only when the `faker` Spring profile is active (local dev). They follow a separate set of rules because they use `runOnChange` or `runAlways` — both of which can re-execute on an existing database.

### How `loadUpdateData` works

`loadUpdateData` is an **upsert** keyed on `primaryKey`:

- Row exists in DB → `UPDATE` only the mapped columns (other columns are untouched)
- Row does not exist → `INSERT` (all mapped columns must be present)

This is different from `loadData`, which only does `INSERT` and fails if the row already exists.

### `runOnChange` vs `runAlways`

| Attribute            | When it re-runs                                               |
| -------------------- | ------------------------------------------------------------- |
| `runOnChange="true"` | Only when the changeset XML or referenced CSV content changes |
| `runAlways="true"`   | Every time the application starts                             |

Use `runOnChange` for older data changesets that have been superseded. Use `runAlways` only for the **latest** data changeset of a table.

### The Latest-Changeset Rule

At any point in time, **exactly one** data changeset per table is the "owner". It is identified by having both `runAlways="true"` and `<delete tableName="..."/>`.

```
older changeset (154020):  runOnChange, no <delete>, maps only its own columns
latest changeset (163520):  runAlways,   <delete>,   maps ALL columns, full dataset
```

The latest changeset always wins: it deletes everything and reloads from its CSV on every app start. This makes the final state predictable regardless of what older changesets did.

### Column mapping rules

| Situation                           | Which columns to map                                 |
| ----------------------------------- | ---------------------------------------------------- |
| Older (non-latest) changeset        | Only the columns introduced by that migration        |
| Latest changeset                    | **All** columns across all migrations for that table |
| Any changeset that inserts new rows | All NOT NULL columns must be present                 |

Mapping only the new column is safe for UPDATE (existing rows), but INSERT of new rows will fail or produce NULL values for unmapped columns.

### Handling foreign key constraints

`<delete tableName="book"/>` will fail if another table has a FK pointing to `book`. Options:

```xml
<!-- Option 1: delete child tables first -->
<delete tableName="order"/>
<delete tableName="book"/>

<!-- Option 2: PostgreSQL — disable FK triggers temporarily -->
<sql>SET session_replication_role = replica;</sql>
<delete tableName="book"/>
<loadUpdateData .../>
<sql>SET session_replication_role = DEFAULT;</sql>

<!-- Option 3: PostgreSQL — cascade (also deletes child rows) -->
<sql>TRUNCATE TABLE book CASCADE;</sql>
```

### runAlways parent/child ordering conflict

**Symptom:** App starts fine the first time, then fails on every restart with:

```
update or delete on table "category" violates foreign key constraint
"fk_book__type_id" on table "book"
```

**Root cause:** Both the parent table faker (category, line 68 in master.xml) and the
child table faker (book, line 149) use `runAlways="true"`. On restart, parent runs first,
tries `DELETE FROM category`, but book still has `type_id` pointing to category → FK violation.

```
Fresh DB (ok):   category faker → DELETE category (empty) ✅ → load ids 1,2,3
                 book faker     → DELETE book (empty)    ✅ → load with type_id 1,2,3

Restart (fail):  category faker → DELETE category ❌  ← book.type_id still references category!
                 book faker     → hasn't run yet
```

**Why `UPDATE book SET type_id = NULL` before the DELETE doesn't work:**

On a fresh DB, the `type_id` column doesn't exist yet at line 68 (it's added at line 148).
The UPDATE fails with "column does not exist" before the DELETE even runs.

**Why listing child tables manually doesn't scale:**

```xml
<!-- BAD: must be updated every time a new table references category -->
<delete tableName="book"/>
<delete tableName="article"/>
<delete tableName="category"/>
```

**Correct fix — use `TRUNCATE ... CASCADE` on the parent faker:**

```xml
<changeSet id="20260610014754-1-data" author="jhipster" context="faker" runAlways="true">
    <sql>TRUNCATE TABLE category CASCADE</sql>
    <loadUpdateData
              file="config/liquibase/fake-data/20260610014754_entity_category.csv"
              separator=";"
              tableName="category"
              usePreparedStatements="true" primaryKey="id">
        <column name="id" type="numeric"/>
        <column name="name" type="string"/>
    </loadUpdateData>
</changeSet>
```

`TRUNCATE TABLE category CASCADE` automatically truncates category and **all tables with FK
pointing to it**, regardless of how many. Safe on fresh DB (table is empty, no FK yet).
Child table fakers that run later (e.g. book faker at line 149) will reload their data normally.

**Why not remove the FK constraint and use a soft reference instead:**

Storing FK values as plain columns (no constraint) means orphaned data is never detected —
`book.type_id = 99` silently returns NULL on JOIN when category 99 is deleted. For billing data,
this means invoices computed against a ghost category with no error raised. Keep real FK constraints
in production; solve faker ordering problems with `TRUNCATE CASCADE`.

### Checklist when adding a new migration with fake data

```
New file (latest):
  [x] runAlways="true"
  [x] <delete tableName="..."/>  inside the faker changeset
  [x] loadUpdateData maps ALL columns (id + all previous + new)
  [x] CSV contains full dataset with all columns

Previously-latest file (now superseded):
  [x] Change runAlways="true"  →  runOnChange="true"
  [x] Remove <delete tableName="..."/>
```

### Why `runAlways` on the latest changeset instead of `runOnChange`

`runOnChange` only triggers when the CSV content changes. If you need a clean reload without touching the CSV, you would have to artificially modify it. `runAlways` removes this friction — the latest data changeset always runs, always produces a predictable state.

### DDL changesets must never use `runOnChange`

`runOnChange` on a `createTable` or `addColumn` changeset will cause Liquibase to retry DDL that already succeeded, resulting in "table already exists" errors. Keep DDL changesets immutable. New columns always go in a new `addColumn` changeset in a new file.

```xml
<!-- WRONG: runOnChange on DDL -->
<changeSet id="..." runOnChange="true">
    <createTable tableName="book"> ... </createTable>
</changeSet>

<!-- CORRECT: immutable DDL, new file for new columns -->
<changeSet id="20260609150432-1">
    <createTable tableName="book"> ... </createTable>
</changeSet>

<!-- In a new file, new changeset -->
<changeSet id="20260609154020-1">
    <addColumn tableName="book">
        <column name="author" type="varchar(255)"/>
    </addColumn>
</changeSet>
```

### Order dependency: data changeset must follow its DDL changeset

The data changeset that maps a column must appear **after** the `addColumn` changeset for that column — either later in the same file, or in a later file included in `master.xml`. Loading a CSV column that does not yet exist in the table will fail.

```
master.xml order:
  20260609150432_added_entity_Book.xml          → createTable (id, name)
  20260609150432_added_entity_Book.xml          → faker: loadUpdateData (id, name)
  20260609154020_updated_entity_Book.xml        → addColumn author
  20260609154020_updated_entity_migrate_Book.xml → faker: loadUpdateData (id, name, author)  ← author exists now
```
